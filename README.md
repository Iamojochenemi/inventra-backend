# Inventra — Multi-Vendor Inventory & Order Management

**Inventra** is a production-grade, multi-tenant backend platform that powers inventory tracking, order fulfillment, payment processing, and delivery logistics for a network of independent vendors — all from a single codebase with strict data isolation.

---

## Project Overview

Inventra solves the core operational challenges of multi-vendor commerce. Each vendor operates its own branches, staff, products, orders, and deliveries, while sharing a unified platform with zero data leakage between tenants. The system is designed for horizontal scalability — add vendors without adding infrastructure.

### Core Capabilities

- **Multi-Tenant Vendor Management** — Vendors, branches, staff roles (owner, manager, inventory, dispatcher, rider), and invitations
- **Inventory Intelligence** — SKU-autogenerated products, branch-level stock tracking, low-stock alerts, audit-logged adjustments
- **Order Lifecycle** — State-machine-gated status transitions (`pending → confirmed → completed/cancelled`) with automatic inventory deduction and restoration
- **Payment Orchestration** — Paystack integration with idempotent webhook handling, invoice generation, and transaction audit records
- **Delivery Logistics** — Rider assignment, status workflow (`pending → assigned → picked_up → in_transit → delivered`), and per-event logging
- **Async Task Pipeline** — Celery workers for email notifications, invoice PDF generation, and vendor invitation emails
- **Analytics Dashboard** — Single-query conditional aggregations for revenue, profit, customer intelligence, delivery metrics, and inventory health
- **Structured Logging & Tracing** — JSON log output with `X-Request-ID` trace propagation across the entire request lifecycle

## Key Engineering Highlights

### Multi-Tenant Database Isolation

Every data-accessing view inherits from `TenantIsolationMixin` (defined in `apps/common/mixins.py`). The mixin automatically scopes all querysets through the `VendorStaff` membership join:

```python
class OrderListView(TenantIsolationMixin, generics.ListAPIView):
    tenant_vendor_field = "vendor"
    # get_queryset() auto-filters: .filter(vendor__staff__user=request.user)
```

This guarantees two critical properties:
- **List views** only return records belonging to the authenticated user's vendors
- **Detail/action views** return **404 (Not Found)** — not 403 — when a cross-tenant resource ID is supplied, preventing data enumeration attacks

### Phase 3 Dashboard Aggregation Optimizations

The analytics dashboard was rewired from 10+ sequential queries to **exactly 5 single-query conditional aggregations** using Django's `Sum`/`Count` with `filter=` kwargs:

| Module | Before | After |
|---|---|---|
| `revenue.py` | 2 queries (Sum + count) | 1 aggregate: all order status counts + revenue |
| `profit.py` | 2 queries (Sum + join aggregate) | 1 aggregate: revenue + COGS via F-expressions |
| `delivery.py` | 3 queries (3 filtered counts) | 1 aggregate: 8 status counts via `Count(filter=Q(...))` |
| `inventory.py` | 2 queries (count + value sum) | 1 aggregate: items, stock value, low-stock, out-of-stock |
| `customer.py` | 1 grouping + 3 count calls | 1 grouping + 3 count calls (ORM constraint) |

Results are cached for 5 minutes via Redis to absorb burst traffic.

### Async Task Pipeline (Celery + Redis)

Time-sensitive operations are dispatched asynchronously to keep API response times predictable:

| Task | Trigger | Queue |
|---|---|---|
| `send_notification_task` | Order status changes, inventory adjustments, delivery events | Redis |
| `generate_invoice_pdf_task` | Successful payment confirmation | Redis |
| `send_vendor_invitation_email_task` | Vendor staff invitation creation | Redis |

All tasks include `max_retries=3` and `autoretry_for` targeting transient network/SMTP failures only — programming errors fail fast.

### Request Tracing & Structured Logging

Every HTTP request is assigned a unique `X-Request-ID` (extracted from the incoming header or generated as UUID4). This ID propagates through:

1. **Request middleware** — sets a `ContextVar` for the current request ID
2. **Logging formatter** — every JSON log line includes `"request_id"` automatically
3. **Response header** — the ID is echoed back in `X-Request-ID` for downstream trace correlation

Log output is single-line JSON, parsable by any log aggregation platform (ELK, Datadog, etc.).

## Local Setup Quick-Start

### Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (with Compose v2)
- Git

### Getting Started

```bash
# 1. Clone the repository
git clone <repository-url>
cd inventra_backend

# 2. Configure environment
cp .env.template .env
# Edit .env with your SECRET_KEY, database credentials, Paystack keys, etc.

# 3. Build and start all services
docker compose up --build

# The API will be available at http://localhost
#   Swagger UI:   http://localhost/api/swagger/
#   ReDoc:        http://localhost/api/redoc/
#   Schema:       http://localhost/api/schema/
```

### Without Docker (Development)

```bash
# 1. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate  # or: venv\Scripts\activate on Windows

# 2. Install dependencies
pip install -r requirements.txt

# 3. Configure environment
cp .env.template .env
# Ensure DB_HOST=localhost, REDIS_CACHE_URL=redis://localhost:6379/1, etc.

# 4. Run migrations
python manage.py migrate

# 5. Start the dev server
python manage.py runserver 0.0.0.0:8000
```

### Environment Variables

All configuration is driven through environment variables. See `.env.template` for the complete reference, including:

| Variable | Description | Default |
|---|---|---|
| `SECRET_KEY` | Django secret key (required) | — |
| `DEBUG` | Debug mode | `False` |
| `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT` | PostgreSQL connection | `inventra`, `inventra_user`, `inventra_pass`, `localhost`, `5432` |
| `REDIS_CACHE_URL` | Redis connection for caching | `redis://localhost:6379/1` |
| `CELERY_BROKER_URL` | Redis connection for Celery | `redis://localhost:6379/0` |
| `PAYSTACK_PUBLIC_KEY`, `PAYSTACK_SECRET_KEY` | Paystack payment gateway keys | — |
| `SENTRY_DSN` | Error tracking (optional) | — |
| `LOG_LEVEL`, `DJANGO_LOG_LEVEL`, `APP_LOG_LEVEL` | Log verbosity | `INFO` |
